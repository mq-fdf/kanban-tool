# 看板任务管理系统 - 面试问题与回答

## 项目概述
这是一个仿 Trello 的看板式任务管理工具，使用 Vue 3 + Pinia + Element Plus + Sass 开发。

---

## 常见面试问题

### 1. 你为什么选择 Vue 3 而不是 React 或其他框架？

**回答：**
选择 Vue 3 主要有几个原因：
- 我之前有 Vue 2 的项目经验，迁移到 Vue 3 比较平滑
- Composition API 提供了更好的逻辑复用能力，相比 Options API 更灵活
- Vue 的单文件组件组织方式让我觉得更清晰
- 生态已经很成熟，Element Plus 和 Pinia 都很好用

不过我也了解 React 的优势，比如更大的生态系统，两者各有千秋。

---

### 2. 在这个项目中，Pinia 相比 Vuex 有什么优势？

**回答：**
Pinia 给我的感觉是更简洁：
- API 更直观，没有 mutations 和 actions 的区分，直接在 actions 里改状态就行
- TypeScript 支持更好，类型推导很方便
- 不需要像 Vuex 那样处理 modules 的嵌套结构
- 代码量更少，同样的功能 Pinia 写起来更短

这个项目一开始我是直接在组件里管理状态，后来觉得太乱了才引入了 Pinia，确实清爽很多。

---

### 3. 拖拽功能是怎么实现的？遇到过什么问题吗？

**回答：**
我用了原生的 HTML5 Drag API，但一开始只支持桌面端。
**遇到的问题：**
- 移动端没有 drag 事件，后来单独加了 touchstart/touchmove/touchend
- 有时候会误触点击任务详情，后来加了 5px 的移动距离判断
- 拖拽过程中计算元素位置有点绕，用了 getBoundingClientRect

**优化方向：**
其实后来想过用 Sortable.js 这种库，但既然已经实现了就懒得改了，下次可能会考虑用库。

---

### 4. 任务按优先级排序是怎么实现的？

**回答：**
很简单，我在 getSortedTasks 函数里给 high/medium/low 分别映射了 0/1/2 的优先级数值，然后用 Array.sort() 排一下。
一开始想在 Pinia 里直接存排序后的数组，但后来觉得在 computed 里排更灵活，虽然每次渲染都会重新排序，但任务量不大，性能影响可以忽略。

---

### 5. 数据持久化是怎么实现的？为什么不用 IndexedDB？

**回答：**
目前就是在每次状态改变时调用 JSON.stringify 存到 localStorage，初始化时再读取。
**为什么不用 IndexedDB：**
- 一开始想试试来着，但 localStorage 已经够用了，数据量不会太大
- IndexedDB 异步回调写起来有点烦，后来就偷懒了
- 如果以后数据量大了或者需要更复杂的查询，可能会换成 IndexedDB 或其他方案

**兼容性问题：**
加了 try-catch 防止 JSON.parse 出错，这时候就用初始数据。

---

### 6. 响应式布局是怎么考虑的？

**回答：**
用了媒体查询：
- 1024px 以下：侧边栏变窄，卡片也小一点
- 768px 以下：侧边栏变成横向的，放在上面

其实移动端的体验还不是很好，特别是拖拽操作，只是勉强能用。如果时间充足的话可能会做个专门的移动端视图。

---

### 7. Element Plus 的组件你是怎么选择和使用的？

**回答：**
主要用了 Dialog、Button、Input、Select、Tag 这些基础组件。
Dialog 用得比较多，新增任务、编辑任务、添加卡片都是 Dialog。
一开始按钮用 type="plain" 报错，后来才知道 Element Plus 的 plain 是属性不是 type 的值。

---

### 8. 如果让你优化这个项目，你会从哪些方面入手？

**回答：**
可以优化的地方还挺多：
1. **性能：**
   - 任务拖拽时可能有不必要的重渲染，可以考虑用 shallowRef
   - 虚拟滚动，如果任务特别多的话

2. **功能：**
   - 添加任务截止时间
   - 任务标签/分类
   - 数据导出/导入
   - 撤销操作

3. **体验：**
   - 更好的移动端适配
   - 拖拽动画再流畅一点
   - 键盘快捷键

4. **架构：**
   - 可以抽一些自定义 hooks，比如 useDrag、useLocalStorage
   - 组件拆分更细一点，把任务卡片拆成单独组件

---

### 9. 你在项目中遇到过最棘手的 bug 是什么？

**回答：**
有个问题困扰了我一下午——关系线的计算有时候会得到 NaN。
一开始找不到原因，后来才发现是因为 DOM 元素还没渲染好就去拿 offsetWidth，得到的是 0，计算时就出问题了。
解决办法：
- 给所有可能为 NaN 的地方加了默认值（|| 280 这样）
- 最后返回前用 isFinite 判断一下，不是有效数字就返回 0

还有一个，拖拽卡片时如果拖出视口会丢失，后来加了边界判断。

---

### 10. 组件之间是怎么通信的？

**回答：**
- 父子组件：主要用 props 和 emit
- 跨组件：都放在 Pinia 里了，比如卡片和任务数据都是全局状态
- 有一些全局事件监听，比如 keydown 监听 Delete 键删除连线

其实可以用 provide/inject，但 Pinia 已经够用了就没折腾。

**回答：**

## 项目亮点

1. **状态管理清晰**：用 Pinia 统一管理，代码不杂乱
2. **响应式支持**：移动端也能用（虽然体验一般）
3. **数据持久化**：刷新不丢失
4. **可视化连线**：展示卡片之间的关系，这点比较有意思
5. **用户体验**：拖拽反馈、优先级排序、进度统计都做了
